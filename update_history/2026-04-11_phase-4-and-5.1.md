## Phase 4 ‚Äî Close the trading loop

### 4.1 Strategy comparison view
New "Compare" button in the strategy modal. Pinned strategies (up to 3, FIFO drop) appear in a side-by-side grid below the saved strategies, showing all 6 score components, expected P&L, sizing, margin, exit plan summary. Top-by-score winner is called out.
- New global state: `_compareList`
- Functions: `toggleCompare`, `isInCompare`, `clearCompare`, `renderCompareDrawer`, `strategyCompareKey`
- Compare button label updates to "‚úì In Compare" when active

### 4.2 Term structure + volatility cone
Two new sections in the IV banner:
- **Term structure**: tiny SVG sparkline showing IV across all available expirations. Labels contango (healthy) vs backwardation (stress).
- **Volatility cone**: current IV vs realized vol percentile bands at 1m / 3m / 6m / 1y lookbacks (computed from chart history). Labels "IV rich vs realized" / "IV cheap vs realized" / "IV fair vs realized".

Both computed inside `analyzeOptionsStrategies` and exposed via `result.termStructure` and `result.volCone`.

### 4.3 Multi-position scenario analysis
New panel under saved strategies with three sliders: Spot Œî%, IV Œî%, Days forward. Each saved position is re-priced via `valueStrategyAtPrice` and the aggregate P&L change is shown. Per-position deltas listed in a row table.
- Functions: `updateScenario`, `renderScenarioPanel`
- Helper in options-analytics.js: `computeScenarioPnL`

### 4.4 Multi-leg roll optimizer
Replaces the placeholder roll suggestions with real candidate-roll math. When a position is approaching its time stop, the modal section calls `loadRollCandidates(strategy)` which:
1. Fetches expirations 5-90 days past current DTE
2. Pulls 2-3 future chains
3. Calls `computeRollCandidates(strategy, futureChains, currentSpot)` which builds 1 candidate per expiration preserving the strategy structure with strikes nearest the original delta-distance from spot
4. Ranks by net debit/credit (credit rolls preferred)
5. Renders top 3 cards showing CREDIT/DEBIT amount, new strikes, new expiration

### 4.5 Margin / buying power impact
New `computeMarginRequirement(strategy, spotPrice)` in `options-analytics.js` with Reg-T formulas for every strategy type:
- Long premium: full debit
- Debit spreads: net debit
- Credit spreads: width √ó 100 ‚àí credit
- Iron Condor: max wing √ó 100 ‚àí credit
- Cash-Secured Put: strike √ó 100 ‚àí credit
- Covered Call / Collar: stock cost basis
- Calendar/Diagonal: net debit
- Ratio spreads: ~20% underlying (naked short approximation)
- Default fallback: max loss

Displayed in the sizing card as a new line: *"Margin: $X/contract ‚Äî $Y total (Z% BP) ‚Äî formula"*

## Phase 5.1 ‚Äî Real-time alerts infrastructure

### Database
New D1 table `position_alerts`:
```sql
CREATE TABLE position_alerts (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  user_id TEXT NOT NULL,
  symbol TEXT NOT NULL,
  strategy_name TEXT,
  trigger_type TEXT NOT NULL,  -- profit_target, stop_loss, iv_rank_above, iv_rank_below, price_above, price_below, time_stop
  threshold REAL NOT NULL,
  legs TEXT,
  status TEXT DEFAULT 'pending',  -- pending, triggered, dismissed
  triggered_at TEXT,
  last_checked TEXT,
  created_at TEXT DEFAULT (datetime('now'))
);
CREATE INDEX idx_alerts_user ON position_alerts(user_id, status);
```

### API endpoint ‚Äî `lite/functions/api/alerts.js`
- `GET /api/alerts?userId=xxx` ‚Äî list user's alerts (most recent 100)
- `POST /api/alerts` ‚Äî create new alert with body `{userId, symbol, strategy_name, trigger_type, threshold, legs}`
- `DELETE /api/alerts?id=N&userId=xxx` ‚Äî delete alert

### SPA UI
- New **üîî Alert Me** button in the strategy modal next to Compare/Optimize/Save
- New **Alerts** section under the saved strategies area showing all pending + triggered alerts with delete buttons
- Functions: `loadAlerts`, `createAlertForStrategy`, `deleteAlert`, `renderAlertsSection`

### Backtester check
Daily routine (`daily-routine.js`) gets a new step **3b** between trade entry and the Friday report:
- Queries all `status='pending'` alerts grouped by symbol
- Fetches each symbol's chain once
- Evaluates `price_above` / `price_below` triggers against current spot
- Updates `status='triggered'` and `triggered_at` on the row when fired
- Updates `last_checked` on every check
- Logs results

For MVP, only `price_above` and `price_below` triggers are evaluated server-side. The other types (profit_target, stop_loss, iv_rank_*) are stored and surfaced in the UI but require chain re-pricing infrastructure that's deferred to a follow-up.

### Email delivery
**Deferred to follow-up.** The MVP surfaces triggered alerts in the SPA when the user opens it. Adding email requires Resend/SendGrid API key + DKIM/SPF setup, which is shippable separately.

## Known caveats
- Server-side alert checks only evaluate `price_above` / `price_below` for MVP. Profit/stop/IV-rank triggers are stored but not yet evaluated by the backtester (need chain re-fetch + per-leg mark-to-market). Follow-up.
- The scenario panel uses an approximate "current spot" derived from the saved strategy's strike midpoint, not the live spot for the underlying. A more precise version would re-fetch per saved position. Acceptable for an MVP "feel" indicator.
- The roll optimizer fetches chains synchronously inside `loadRollCandidates`, so opening a modal with rolls available adds ~500ms latency. Acceptable.
- The compare drawer's per-card delete button uses inline JSON.stringify of the strategy object, which is fragile for strategies with nested data. Would refactor to use the compare key instead in a follow-up.
